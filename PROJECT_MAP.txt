3. BACKEND — MODELS (Database Schemas)
================================================================================

--- CORE DOMAIN ---

FILE: backend/models/Order.js (859 lines) ★ LARGEST & MOST IMPORTANT MODEL
PURPOSE: Central order entity — single document contains ALL order context
EMBEDDED SCHEMAS:
  - customerSchema: buyer name, phone, email, GSTIN
  - addressSchema: delivery/pickup with pincode validation
  - productSchema: items, quantity, pricing, HSN, SKU
  - packageSchema: weight, L×W×H, volumetric calc
  - paymentSchema: mode (Prepaid/COD/Pickup/REPL), amounts
  - delhiverySchema: waybill, tracking URL, pickup status, cancellation
  - mpsSchema: multi-package shipment tracking
  - ndrSchema: NDR info (★ SINGLE SOURCE OF TRUTH for NDR — not NDR.js)
  - billingInfoSchema: zone, weight (grams), charges breakdown, billing status
KEY INDEXES: user_id+status, user_id+order_date, delhivery_data.waybill (unique)
VIRTUALS: total_products, total_products_quantity
PRE-SAVE HOOKS: volumetric weight calc, status history tracking
INSTANCE METHODS: canBeCancelled(), isInTransit(), updateStatus()
STATIC METHODS: getOrdersByStatus(), getOrdersCountByStatus()
WHEN TO LOOK HERE: Any order-related feature, status changes, billing, NDR

FILE: backend/models/User.js (578 lines)
PURPOSE: Seller/business accounts with KYC, API keys, wallet, documents
KEY FIELDS: user_type (e-commerce/D2C/manufacturers/enterprise/courier/individual),
  company_name, GSTIN, phone/email (unique), bank_details (encrypted),
  KYC status + verification history, API keys (encrypted), wallet_balance,
  account_status, user_category (New User/Basic/Lite/Advanced)
KEY INDEXES: phone_number, client_id, api_details.public_key
PRE-SAVE: client_id generation (SS#####), API key generation, password hashing
INSTANCE METHODS: comparePassword(), createPasswordResetToken(),
  createEmailVerificationToken(), OTP management (create, verify, increment, reset)
STATIC: findByEmailOrPhone()
VIRTUAL: can_create_shipments (based on KYC + account status)
WHEN TO LOOK HERE: Auth, registration, KYC, wallet, user categories, API keys

FILE: backend/models/Customer.js (287 lines)
PURPOSE: End-customer (buyer) database — many customers per seller
KEY FIELDS: user_id (ref to User), name, phone, email, address, channel,
  order statistics, tags, social links
STATICS: findOrCreate(), getCustomersByUser(), getCustomerStats()
WHEN TO LOOK HERE: Customer management, order creation (buyer info)

FILE: backend/models/NDR.js (430 lines) ⚠ UNUSED BY ROUTES
PURPOSE: Standalone NDR model for future advanced features
NOTE: Routes query Order.ndr_info instead. This model has customer comms,
  agent assignment, auto-resolution rules — none currently wired up.
WHEN TO LOOK HERE: Planning future NDR features (not current functionality)

FILE: backend/models/Carrier.js (152 lines)
PURPOSE: Shipping carriers/service types (Delhivery Surface, Air, DTDC, etc.)
KEY FIELDS: carrier_code (unique, e.g. 'DELHIVERY_SURFACE'), display_name,

FILE: backend/services/rateCardService.js (469 lines)
PURPOSE: Calculate shipping charges (weight × zone × category × carrier)

  getRateCard()                    → Fetch rate card (5-min cache)
  calculateShippingCharges()       → Legacy (Surface default)
  calculateShippingChargesByCarrier() → New (carrier-specific)

WEIGHT: All stored in GRAMS internally; volumetric = L×B×H/5000 kg → grams
ZONES: A-F standard (or regional: City/Regional/Metro/RestOfIndia/SpecialZone)
COD: Percentage + minimum + 18% GST if flag set
WHEN TO LOOK HERE: Pricing bugs, rate card changes, charge calculation logic

  processSorterImageWebhook()    → Handle weight images
  processQCImageWebhook()        → Handle QC images

16. KEY DATA FLOWS & ARCHITECTURE
================================================================================

--- ORDER CREATION FLOW ---
1. User fills OrderCreationModal (frontend)
2. Frontend validates, calls shippingService.calculateShippingCharges()
3. POST /api/orders → auth middleware → validate input
4. rateCardService.calculateShippingChargesByCarrier() (zone from Delhivery API)
5. Create Order document in MongoDB
6. delhiveryService.createShipment() → Delhivery API → waybill + label_url
7. Update Order.delhivery_data with response
8. labelRenderer generates shipping label
9. emailService sends confirmation
10. Return order with waybill to frontend

--- NDR WORKFLOW ---
1. Delhivery webhook → /v1/delhivery/scan-status (webhooks.js)
2. webhookQueue enqueues for async processing
3. webhookService.processScanPushWebhook() maps status to 'ndr'
4. Updates Order.status = 'ndr'
   ⚠ Does NOT populate Order.ndr_info (known gap in v1 webhook)
5. User sees NDR in NDR.tsx → selects RE-ATTEMPT or PICKUP_RESCHEDULE
6. POST /api/ndr/:order_id/reattempt → ndr.js route
7. delhiveryService.updateNDR() → Delhivery POST /api/p/update {waybill, act}
8. Returns UPL ID → polls checkNDRStatus()
9. Updates Order.ndr_info.resolution_action

--- WALLET RECHARGE FLOW ---
1. User clicks recharge → RechargeModal (frontend)
2. walletService.initiatePayment() → POST /api/users/wallet-recharge
3. hdfcPaymentService.createSession() → HDFC SmartGateway
4. User completes payment on HDFC page
5. Callback → captureHdfcCallback script / route handler
6. Transaction created, wallet_balance updated
7. PaymentConfirmation page displayed

--- WEBHOOK PROCESSING ---
1. Delhivery sends scan/EPOD/sorter/QC data to /api/webhooks/*
2. webhookAuth validates X-API-Key
3. v1 path: webhookQueue.enqueue() for async processing
4. webhookService maps status → updates Order + tracking entities
5. websocketService broadcasts real-time update to connected clients

--- ADMIN IMPERSONATION ---
1. Admin selects client → adminService.impersonateClient()
2. Returns impersonation token
3. Opens user dashboard with token in URL params
4. AuthContext reads token, acts as that user

--- BILLING CYCLE ---
1. Orders accumulate in 15-day billing cycle (BillingCycle model)
2. Cycle closes at period end (1st-15th or 16th-end)
3. invoiceService generates Invoice with GST calculation
4. Invoice available for download via /api/billing/invoices
